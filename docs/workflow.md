# OPSX + Superpowers Workflow Guide

A general overview of how features and fixes move through this repo. The authoritative rules live in the OpenSpec schema and the `/opsx:*` command files — this document is the map, not the code.

---

## The four stages

```
/opsx:explore  →  /opsx:propose  →  /opsx:apply  →  /opsx:archive
   (Think)         (Plan)            (Build)         (Close)
```

Each stage is a slash command in `.claude/commands/opsx/`. Every change passes through all four.

| Stage | Command | Produces |
|---|---|---|
| **Explore** | `/opsx:explore` | A draft `design.md` in `openspec/changes/_draft/<topic>/` (gitignored) |
| **Propose** | `/opsx:propose` | The full change directory: `proposal.md`, `design.md`, `specs/`, `tasks.md` |
| **Apply** | `/opsx:apply` | Working code, tests, commits, and dev-log entries |
| **Archive** | `/opsx:archive` | Archived change in `openspec/changes/archive/`, synced specs, lessons file |

---

## Where the rules live

There is **one source of truth** for artifact content and apply-time discipline: the project-local schema at `openspec/schemas/openspec-superpowers/schema.yaml`.

| Rule type | Lives in |
|---|---|
| What `proposal.md` / `design.md` / `specs/` / `tasks.md` must contain | Schema → each artifact's `instruction:` block |
| TDD pattern, code-review checkpoints, checkbox discipline, parallel dispatch, dev-log update | Schema → `apply.instruction` |
| Orchestration (which skills to invoke, user prompts, file moves) | `.claude/commands/opsx/*.md` |
| Project-wide facts (stack, conventions, test commands) | `CLAUDE.md` and `openspec/config.yaml`'s `context:` block |

If a rule disagrees between the schema and a command file, the schema wins.

## Stage 1 — Explore

**Purpose:** Think through an idea without committing to a name or producing real artifacts yet.

**What happens:**
- The conversation roams: clarifying questions, codebase exploration, comparison tables, ASCII diagrams.
- When the shape settles, a draft `design.md` is written to `openspec/changes/_draft/<topic>/design.md`.
- `superpowers:brainstorming` is invoked as a **REVIEWER** of that draft — it surfaces gaps, contradictions, scope drift.
- If the change has UI surface, Visual Companion is offered (optional — only when visual decisions benefit from side-by-side mockups).
- The user iterates on the draft until satisfied.

**What does NOT happen:** No code is written. No `openspec new change` is run. The kebab-case change name is deferred.

The draft is **gitignored** — it lives outside source control until `/opsx:propose` promotes it.

---

## Stage 2 — Propose

**Purpose:** Promote the draft into a real change and generate the rest of the artifacts.

**Flow:**
1. Find the draft in `openspec/changes/_draft/`. If multiple, ask the user to pick.
2. Ask the user for the final kebab-case `<name>`.
3. Run `openspec new change <name>` to scaffold `openspec/changes/<name>/`.
4. Move `_draft/<topic>/design.md` → `<name>/design.md` and remove the empty draft dir.
5. **STOP and confirm** with the user before generating remaining artifacts.
6. Generate in dependency order: `proposal.md` → `specs/<capability>/spec.md` → `tasks.md`.

Each artifact is generated by calling `openspec instructions <artifact> --change <name> --json` and following the returned `instruction:` and `template:` fields.

**Cold-start fallback:** If no draft exists, the user is offered three options: (1) run `/opsx:explore` first (recommended), (2) brainstorm in-line now (legacy), or (3) skip brainstorming and describe directly.

---

## Stage 3 — Apply

**Purpose:** Implement the tasks in `tasks.md` with TDD discipline and review checkpoints.

**TDD discipline (Iron Law):** No production code without a failing test first.

Every `## N` task group follows this structure:

```
## N. Feature name

- [ ] N.1 RED   — write failing test for <behavior> → run → confirm FAILURE
- [ ] N.2 GREEN — minimal impl to pass the test
- [ ] N.3 GREEN — additional impl if needed
- [ ] N.Z   Run superpowers:requesting-code-review on the diff for group N;
              address CRITICAL/HIGH findings before moving on.
- [ ] N.Z+1 Update docs/log/YYYY-MM-DD.md — commit hash, feature bullets,
              review findings, test count, TDD evidence (paste RED failure
              lines for each new test).
```

The **final group** adds two tasks before `N.Z` whenever the UI is touched:

```
- [ ] M.J Write/update Playwright E2E test under e2e/ for the affected flow.
- [ ] M.K Run superpowers:verification-before-completion (full backend +
          frontend test suites, console.log/println greps, diff review).
```

**Parallel dispatch:** When 2+ task units within a group touch disjoint files, they're marked `[parallel]` and a `N.0` task dispatches `superpowers:subagent-driven-development` to run them concurrently.

**Checkbox discipline:** Mark each task `[x]` *immediately* after finishing — never batch updates. Use the Edit tool (content-matched), not `sed` (line-numbered, fails silently when lines shift).

**Commit per group:** Each `## N` group ends with one commit containing test + implementation + dev-log update.

---

## Stage 4 — Archive

**Purpose:** Close out a completed change.

**Flow:**
1. Confirm all artifacts and tasks are `done`. Warn (don't block) on incompletes.
2. Compare each delta spec in `openspec/changes/<name>/specs/` against the corresponding main spec in `openspec/specs/`. Offer to sync via `openspec-sync-specs`.
3. Move the change directory to `openspec/changes/archive/YYYY-MM-DD-<name>/`.
4. Draft a per-change **lessons file** at `docs/lessons/YYYY-MM-DD-<name>.md` covering:
   - **Scope drift** — what shipped vs. what was proposed.
   - **Recurring review findings** — patterns across group `N.Z` checkpoints.
   - **TDD observations** — RED-phase gotchas, untagged failing tests, baseline issues.
   - **Surprises** — anything future similar work should anticipate.

   The draft is shown to the user — **never auto-written**. The user reviews/edits before it's written to disk.

5. Optionally update `README.md` — only on explicit user "yes".

## Documentation outputs

| File | Lives at | Written by | Cadence |
|---|---|---|---|
| Dev log | `docs/log/YYYY-MM-DD.md` | `/opsx:apply` (during each group's `N.Z+1` task) | One entry per task group |
| Lessons | `docs/lessons/YYYY-MM-DD-<name>.md` | `/opsx:archive` (user-reviewed) | One file per archived change |
| Spec | `openspec/specs/<capability>/spec.md` | `/opsx:archive` (sync step) | Updated on each archive that touches the capability |

The dev log captures the *play-by-play* of implementation; lessons capture *what to remember next time*; specs capture *what the system does now*.

## Quick reference

| Question | Answer |
|---|---|
| Where does the design live during exploration? | `openspec/changes/_draft/<topic>/design.md` (gitignored) |
| When is the final change name decided? | At `/opsx:propose` — the user is asked |
| What enforces the TDD/checkpoint/checkbox rules? | The schema's `apply.instruction`, returned by `openspec instructions apply` |
| What's the difference between `proposal.md` and `design.md`? | Proposal = **why** (for human readers). Design = **how** (architecture, decisions, alternatives). |
| Are tests committed before they pass? | No. Only GREEN tests are committed. RED failure output goes to the dev log as TDD evidence. |
| How do `[parallel]` tasks work? | A `N.0` task dispatches `superpowers:subagent-driven-development` to run disjoint units concurrently |
| Where do I look up project conventions? | `CLAUDE.md` (project-wide facts) + `openspec/config.yaml` `context:` (injected into every artifact prompt) |
