### Commits

- `640cba7` feat(messaging): publish MessagePostedEvent after each message insert

### Features delivered

- **`MessagePostedEvent`**: Java record `(MessageThread thread, Message message)` published from `MessagingService` after each message insert, within the same `@Transactional` boundary.

- **`MessagingService` event publishing**: All 4 write methods (`createThreadAsAdmin`, `createThreadAsClient`, `postAdminReply`, `postClientReply`) now publish `MessagePostedEvent` via `ApplicationEventPublisher` after `messageRepo.save()`.

- **`AppProperties`**: `@ConfigurationProperties(prefix = "app")` class reading `app.publicBaseUrl` from `application.yml` (default: `http://localhost:4200`).

- **`MessageNotificationListener`**: `@TransactionalEventListener(phase = AFTER_COMMIT)` component. On admin message: emails the client's address. On client message: emails the firm's notification mailbox (`MailProperties.notificationEmail`). Skips silently if client has no email. Catches `MailException` and logs at ERROR without propagating. Dev environment uses `LoggingMailSender` (logs instead of sending).

- **E2E spec**: `e2e/admin-client-messaging.spec.ts` — full admin↔client roundtrip using mocked API routes. Covers: admin creates thread, client sees unread badge, client replies, admin sees badge and reply. Uses the project's fake JWT cookie + `page.route()` mock pattern.

### E2E

### Deferred to v2

- **Admin-side "awaiting client" indicator.** Today the admin only sees a badge when the client has sent an unread message. After the admin sends, the thread visually goes silent — no signal that the ball is in the client's court. Idea: surface a per-thread chip on the admin thread list — `Awaiting client` (admin sent, `client_unread_count > 0`), `Client read` (admin sent, `client_unread_count = 0`), `Unread` (current behavior). Requires adding `clientUnreadCount` and last-sender type to `MessageThreadSummaryDto`.

- **Portal messaging UX redesign — dashboard-card-as-primary-inbox.** Move messaging out of the top navbar and surface it primarily through the dashboard's right-side Messages card (which is currently mock data). Final v2 design:
  - **Dashboard Messages card** (`dashboard.component.ts:55-59` / `.html:54-87`): replace the hardcoded `messages: PortalMessage[]` array with a signal populated from `PortalMessagesService.listThreads()`. Show the top 3 most recent threads. Each row navigates to `/portal/messages/:threadId`. Keep the existing visual styling (`.msg-row`, `.msg-dot`, `.msg-title`, `.msg-sender`, `.msg-meta`). Sender label is `"Your accountant"` for ADMIN-sent threads, the date is the relative `lastMessageAt`. Unread state on the row drives the `.unread` class.
  - **Card footer**: add a `View all messages →` link at the bottom of the card that navigates to `/portal/messages`. The dedicated inbox page stays as the overflow target (full thread list when there are more than 3).
  - **Navbar**: remove the standalone `Messages` link + badge (currently in `navbar.component.html` / `.ts`). Replace the global cross-page signal by **adding an unread badge to the existing `Dashboard` nav link** (e.g., `Dashboard [3]`) — no new nav element, just a signal on the existing link. Source for the count: `PortalMessagesService.getUnreadCount()` (already wired).
  - **Reply UI is unchanged**: `/portal/messages/:threadId` and `PortalThreadViewComponent` remain the canonical place to read and reply. The dashboard card and the full inbox are both navigation surfaces into the same thread view.
  - **Tests to update**: `navbar.component.spec.ts` (Messages-link tests removed; Dashboard-link badge test added), `dashboard.component.spec.ts` (mock data tests replaced with service-backed tests), and possibly add a new spec for the "View all" navigation.

- **Role-aware login redirect + drop the navbar "Admin" button.** Today every OAuth2 login lands at `/portal/dashboard` regardless of role (`OAuth2SuccessHandler.java:34, 75`), and the navbar shows an `Admin` link to ADMIN users (`navbar.component.html:23-25`). For v2, admins should land directly in the admin UI and not see the client portal at all.
  - **Backend**: in `OAuth2SuccessHandler.onAuthenticationSuccess()`, branch on `user.getRole()` and redirect ADMIN to the admin landing page (`http://localhost:4200/admin/clients`) and USER to the current `/portal/dashboard`. Replace the single `app.oauth2.redirect-uri` property with two: `app.oauth2.redirect-uri.admin` (default `http://localhost:4200/admin/clients`) and `app.oauth2.redirect-uri.user` (default `http://localhost:4200/portal/dashboard`).
  - **Frontend navbar**: delete the `Admin` link block at `navbar.component.html:23-25` (the `@if (currentUser()?.role === 'ADMIN')` Admin button). Update the `navbar.component.spec.ts` admin-link test to assert the link is **no longer rendered for ADMIN**.
  - **Brand-logo behavior** (subtle UX): once the Admin button is gone, give admins a way back to `/admin/clients` if they navigate away (e.g., click the GWH Accounting logo). Either make the brand link conditional on role (`/admin/clients` for ADMIN, `/` for USER), or leave the logo always pointing at `/` and accept that admins use the URL bar. Pick one in v2 — leaning toward the conditional brand link since admins shouldn't see public-site content.
