# Dev Log — 2026-06-17

## opsx-security-review-gate

**What shipped:** The built-in `/security-review` command is now wired into the OpenSpec apply workflow as a mandatory final gate.

**Key changes:**
- `openspec/schemas/superpowers-driven/schema.yaml` (`tasks.instruction`) — the final task group now mandates two ordered tasks: `Run /security-review on the branch diff` immediately before `superpowers:verification-before-completion`. Every `tasks.md` generated by `/opsx:propose` carries the security task automatically.
- `.claude/commands/opsx/apply.md` — new step-3 dispatch bullet ("Final group's security-review task") invokes the `security-review` skill on the branch diff, reads findings by severity inline (no JSON parser), and **blocks on Critical/High/Medium** (pause + 3 options: fix/resume, waive with justification, abort); **Low is advisory**. One guardrail bullet added.

**Decisions:**
- Built-in `/security-review` chosen over the ECC checklist skill (wrong stack — Next.js/Supabase flavored) and over a `superpowers:` skill (none exists).
- Final gate only (whole-branch diff), not per-group — matches the command's "pending changes on current branch" semantics; avoids redundant reviews.
- Security runs before verification so security fixes get re-tested by verification-before-completion.
- Block threshold raised to Critical/High/Medium per user decision (Low advisory only).

**Verification:** Markdown-only change — no pytest/vitest surface. Two `python3` assertions confirm (a) the schema instruction orders security before verification, (b) apply.md contains the dispatch rule, block message, guardrail, and correct ordering. Dispatch logic dry-run confirms Critical/High/Medium → pause, Low-only → continue.

**Notes:**
- A subagent self-modification security warning fired when editing `apply.md` — false alarm; editing the apply command IS the user-approved task.
- Spec/design/plan: `docs/superpowers/specs/2026-06-17-opsx-security-review-gate-design.md`, `docs/superpowers/plans/2026-06-17-opsx-security-review-gate.md`.

## admin-client-export

**What shipped:** Admins can now multi-select clients from the client list and export their metadata (CSV), documents (ZIP), or both (combined ZIP).

**Key changes:**

Backend (`com.gwhaitech.accountingfirm.client`):
- `AdminExportService` — validates ownership (server-enforced `admin_id` check), streams CSV via `PrintWriter`, streams ZIP via `ZipOutputStream` matching the existing `MeDocumentService.zipForYear()` pattern. 200-client hard cap.
- `AdminExportController` — `POST /api/clients/export` (metadata/documents/both) and `GET /api/clients/ids` (lightweight IDs-only endpoint for select-all across pages).
- `ExportValidationException` + handler in `GlobalExceptionHandler` → 400 responses.

Frontend (`src/app/features/admin/clients/`):
- `AdminClientsComponent` — checkbox column, select-all button, 200-cap enforcement with inline message, Angular `effect()` clears selection on filter change, export toolbar (conditional on `selectedClientIds().size > 0`) with "Clear" deselect-all button, `isExporting` signal drives button disabled/spinner state.
- `AdminExportDialogComponent` — standalone Material dialog: include metadata / include documents checkboxes (both checked by default), year selector (last 10 years + "All years", shown only when documents checked), Cancel/Export actions.
- `AdminExportService` — `getAllClientIds()`, `downloadExport()` (blob POST → Content-Disposition filename parsing → `URL.createObjectURL` → anchor click → revoke; blob error body parsed for server message).

**Decisions:**
- Dedicated `AdminExportService` (not bolted onto `ClientService`) per D2.
- Direct CSV download (no ZIP wrapper for metadata-only) per user decision.
- `{ClientName}-{clientId}/` folder naming in ZIP guarantees uniqueness without collision detection.
- Select-all fetches all matching IDs in one round trip via `GET /api/clients/ids` rather than paginating.
- Error messages surface from server JSON body (`err.error.message`) not generic strings.

**Verification:** Backend 20/20 tests pass. Frontend 36/38 pass (2 pre-existing badge tests). E2E 6/6 pass (toolbar, select-all, filter clears, dialog, CSV download, cancel). Security review: no Critical/High/Medium findings.
