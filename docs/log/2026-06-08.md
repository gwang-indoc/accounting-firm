# Dev Log — 2026-06-08

## locale-switch-refactor — archived

**What shipped:** Language preference now persists per authenticated user. Toggling EN/ZH while logged in saves the choice to the backend; logging in on a new device restores it immediately.

**Key changes:**
- V11 Flyway migration: `ALTER TABLE users ADD COLUMN language VARCHAR(10) DEFAULT NULL`
- `PATCH /api/auth/me/language` — saves language for the authenticated principal (204)
- `GET /api/auth/me` — now includes `language` field in `UserDto`
- `TranslationService.applyProfileLanguage(lang)` — called after `loadCurrentUser()` resolves; profile wins over localStorage on conflict; pushes localStorage value to profile if profile is null
- `setLanguage()` — fires a background PATCH when authenticated (fire-and-forget, silent on failure)
- `APP_INITIALIZER` consolidated to one `initApp` function to eliminate the race between init and auth load

**Evaluator notes:**
- Group 1 passed first try (96.4). Minor: no validation that `language` is `en` or `zh` — accepted tradeoff for now.
- Group 2 needed two passes. First pass caught: missing `await` on `loadLanguage()` inside `applyProfileLanguage`, a race condition between two parallel `APP_INITIALIZER` entries, and DI cascade from `AuthService→TranslationService`. Fixed by making the call async, combining initializers, and removing TranslationService from AuthService (replaced with `isAuthFn` callback).

**Tests:** Backend +4 tests (AuthControllerTest). Frontend +6 tests (translation.service.spec.ts). Pre-existing baseline failures unchanged (14 frontend files, 3 backend tests).
